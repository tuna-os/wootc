● Good challenge — to be clear, decryption is not part of the product flow; it's
  a one-time salvage of tonight's test VM. Three things got tangled here:

  1. What the spec says (§3.5): wootc never decrypts a user's C:. On BitLocker
  machines, root.disk goes on a separate unencrypted partition (auto-created D:,
  or one the user picks). That's exactly the design you remember, and deploy.sh
  already supports it — it scans every NTFS partition for
  /wootc/disks/root.disk, so a root.disk on an unencrypted D: is found the same
  way.
  2. What happened in this test: the E2E is meant to exercise Normal mode — a
  plain unencrypted C:. But Windows 11 24H2's OOBE silently turned on device
  encryption (TPM + Secure Boot machine), which the test never accounted for. So
  the VM ended up in a state the test didn't intend: root.disk already sitting
  on a now-encrypted C:.
  3. Why decrypt rather than repartition tonight: the clean fix —
  PreventDeviceEncryption in autounattend — is already committed, so every
  future test install comes up unencrypted and this never recurs. For this VM,
  the choice was (a) reinstall Windows (~30 min), (b) shrink C: and move
  root.disk to a new D: (~similar time, and changes what the current run
  asserts), or (c) decrypt in place. Decryption was already underway and is at
  ~13% remaining ("Protection Off" meant it was encrypted with a clear key —
  pure ciphertext overhead, protecting nothing), so it's the fastest path to a
  phase-2 attempt tonight.

  Worth adding later: a second E2E variant that deliberately keeps BitLocker on
  and exercises the D:-partition path, since that's what real Win11 machines
  will look like out of the box. Want me to file an issue for that once this run
  completes?

  Decryption should finish in a few minutes; the monitor will wake me and I'll
  re-arm the one-shot and boot the deployer.
